### 1단계 - 설계결정문서
- [x]  **왜 청크 크기 800 / min 350인가?** 배달 정책 문서가 조항 단위로 이미 쪼개져 있다는 사실을 근거로 설명. “블로그 글 / 장문 PDF” 같은 다른 도메인이라면 어떻게 바꿀 것인가?  
    현재 사용되는 배달 정책 문서(md)의 경우, category(쿠폰,취소,환불,배달 등)별로 이미 쪼개져 있는 상태. 청크 크기를 800 으로 하는 경우, 위의 RAG 시드 로그를 확인했을 때 보이듯이 각 문서가 모두 하나의 Chunk 로 저장됩니다.  
    카테고리 별로 분류도 잘 되어있고, 문서를 더 잘게 나누어야 할 필요성이 없는 경우이기 때문에 문서 당 하나의 Chunk 가 되게끔 Chunk Size 를 셋팅했습니다.  
    추가적인 가설이지만 기본 "TokenTextSplitter" 가 아닌 "Seperator" 를 지정할 수 있게 한다면 "##" 등의 구분자로 구분이 가능할 것으로 보이나 이 경우, 나누어진 문서 내용에 대한 Metadata 가 프롬프트에 넘어가지 않기 때문에 더 정확도가 떨어질 것으로 예상됩니다.  
        ex. "QuestionAnswerAdvisor" 는 Document 의 Metadata 를 프롬프트에 추가하지 않음. 단순히 문서의 내용만을 추가하기 때문에 문서 하나당 하나의 Chunk 인 경우, 첫문장이 자연적으로 추가되기 때문에 문제가 없지만, 더 작은 Chunk 로 나눠지는 경우, 첫 문장이 빠지게 되면서 분류가 더 어려워질 것으로 예상됩니다.  
      
    장문 글을 저장한다면 Chunk 를 더 작은 단위로 나누고, 테스트로 어느 정도의 OverLab 을 적용할 것인지를 결정할 거 같습니다.  
    추가적으로 Chunk 마다 글의 분류값이나 제목, Tag 값을 metadata 로 추가하고 프롬프트에 추가될때 메타데이터도 같이 적히도록 변경할 것 같습니다.  
- [x]  **왜 Top-K=4인가?** 정책 문서가 7건인 상황에서 1 / 4 / 10 중 4를 선택한 근거를 설명  
    일단 저는 Top-K 를 3으로 변경했습니다.    
    그 이유는 위의 테스트에서 최대 사용되는 문서의 개수가 2개였던 점 + 문서별 분류가 확실하고, 각 분류단위로 최대 2개의 문서가 존재하는 점을 근거로 삼았습니다.    
    배달지연+환불 혹은 환불+취소 의 경우, 겹치는 부분이 생길 수 있을 것으로 판단되어 최대 3개로 잡았습니다.  
      
    top-k 값은 결국 최종 LLM 에게 전달한 문서의 개수를 정의합니다. 문서의 개수를 많이 주는게 꼭 좋은 응답을 보장하진 않고, 비용 증가는 확실하기 때문에 굳이 증가할 필요가 없다고 생각됩니다.
- [x]  **왜 `QuestionAnswerAdvisor.order(20)`인가?** `memoryAdvisor(10) → ragAdvisor(20) → performanceAdvisor(100)` 순서가 필요한 이유를 **시나리오 5를 예시로** 설명  
    현재 기준에서 Order 가 중요하다고 생각되지 않습니다.  
    현재 코드 기준으로 모델에게 넘어가는 프롬프트를 보면 RAGAdvisor 에게 이전 Memory Context 가 전달되지 않습니다. 이전 메모리 Context 는 최종 LLM 에게만 전달됩니다. 그렇다면 현재 기준으로 Order 는 큰 의미가 없다고 생각됩니다.  
    만약 프롬프트 생성 부분을 변경해서 이전 Context 가 RAGAdvisor 에게도 넘어가게 변경한다면 의미가 있을 것으로 보입니다.  
- [x]  **`similarityThreshold=0.5`의 근거.** 너무 낮으면 / 너무 높으면 각각 어떤 문제가 생기는가? (실험은 3단계 과제)
    너무 낮은 경우, 관련도가 떨어지는 문서까지 모두 검색되어, 정확도를 떨어뜨리고 토큰 사용량을 증대시킵니다.  
    너무 높은 경우, 사용자의 문의에 대해 유연성이 떨어집니다. 예를 들어 정확히 "배달 지연"이라는 문구가 없는 경우 문서가 검색되지 않을 수 있습니다.

- `Prompt 의 getUserMessage` 코드를 살펴봤습니다.  

```java
    public UserMessage getUserMessage() {
        for (int i = this.messages.size() - 1; i >= 0; i--) {
            Message message = this.messages.get(i);
            if (message instanceof UserMessage userMessage) {
                return userMessage;
            }
        }
        return new UserMessage("");
    }
```

- message 를 반환하는 부분에 보면 기존 messages 를 모두 append 해서 return 하는게 아닌 UserMessage 1건만 반환합니다.

- [x]  **반대 순서가 더 나은 상황이 존재하는가?** 예를 들어 “Memory에는 개인 정보가 들어있어 임베딩하면 안 되는 경우”를 고려해 보라 (Round 5 Guardrail 예고)
    제 테스트에서는 반대로 결과가 나왔기 때문에 반대의 경우를 고민해보겠습니다. (Memory 가 임베딩되어야 하는 경우)  
    테스트에서와 같은 단순 "이전 주문번호가 필요한 경우" 는 해당되지 않습니다. 주문번호는 관련 문서를 탐색하는데 영향을 끼치지 않습니다.  
    이전 대화가 "오늘 날씨 상태에 대해 대화했고" 최종 질문이 "그럼 지금은 환불 가능해?" 라는 방식으로 문의했다면 이전 대화 이력도 임베딩 하는게 정확도를 높일 수 있다고 생각됩니다.  
    실제 테스트 시에 RAG 결과 검색되는 문서는 `"# 기상 악화 시 배달 지연 안내"` 가 아닌 `"# 주문 취소 정책"` 였습니다.
