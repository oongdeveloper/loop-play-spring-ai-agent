### Issue 1. "Structured Output"이 필요한 프롬프트와 "content"만으로 동작해도 되는 프롬프트가 충돌함.

**문제**
`주문` 관련 단순 문의에도 `구조화된 응답`이 반환됨.
```
summary: 현재 주문이 배달 중이 아니라면 취소가 가능합니다.
category: CANCEL
urgency: 정상
affectedParty: PLATFORM
nextAction: 주문 상태 확인 후 고객에게 안내
requiresHumanVerification: false
neededInfo: []
```

**원인**
프롬프트에 응답 포맷에 대한 설명이 있어, 구조화된 포맷으로 응답.

**해결**
구조화된 응답용 프롬프트와 단순 응답용 프롬프트를 별도로 분리

### Issue 2. [규칙] 과 충돌하여 Tool 호출이 되지 않음. (시나리오 2)

**문제**
`getOrderDetail` 이 호출되지 않아 원인 분석. 프롬프트 전문 + Tool 정보를 AI 로 분석한 결과,
`[규칙]` 의 "정보가 부족하면 되묻는다" 항목과 Tool 사용 지시가 충돌하는 것을 발견.

**원인**
주문번호가 명확히 제공되었음에도 LLM 이 "정보 확인이 필요하다" 고 판단하여 Tool 호출 대신 되묻는 응답을 반환.

**시도한 방법**
- `@Tool` description 에 "반드시 호출하라" 명시 → Tool 호출 됨 ✅
- 프롬프트 규칙 수정 → Tool 호출 됨 ✅

**최종 적용**
```
Tool Description 에 "반드시 호출하라" 로 둘 경우, 다른 Tool 과 혼동을 줄 수 있어, 시스템 프롬프트에 내용 추가 
Tool Description 을 변경하고, 다른 Tool 이 수행되는지 3회 정도 테스트 시에는 문제가 없었지만, '반드시' 를 명시하긴 위험부담이 있다고 생각됨.
- 주문번호가 명확히 제공된 경우, 되묻지 않고 바로 Tool을 호출한다.
```

### Issue 3. 취소 사유(reason) 누락 시 Tool 미호출. (시나리오 3)

**문제**
취소 사유를 찾을 수 없는 경우 Tool 이 호출되지 않음.

**해결**
`@ToolParam` description 에 기본값 힌트 추가 후 Tool 호출 정상 동작.

**시도한 방법**
- `required = false` 처리 → Tool 호출 안 됨
- description 에 기본값 명시 → Tool 호출 됨 ✅

**최종 적용**
```java
@ToolParam(description = "취소 이유. 고객이 언급한 취소사유를 그대로 사용한다. 명시적인 사유가 없으면 '고객 요청'으로 입력한다.")
```

## 설계 결정문서
- [x]  `OrderDetailView`는 내부 `Order` 클래스의 필드 중 **무엇을 의도적으로 뺐는가?** (예: `deliveryAddress`, `canceledReason`, `riderLocation` 중 일부) 그 이유는?
   > `deliveryAddress` 컬럼 값 삭제.  
   > DB → @Tool 반환값 → LLM 프롬프트에 포함 → LLM 처리 → 최종 응답  
   > deliveryAddress 는 LLM 에게 사용자의 개인정보가 그대로 넘어갈 수 있음.
- [x]  `@Tool`의 `description`을 **한국어로 썼는가, 영어로 썼는가?** 어떤 기준으로 결정했는가?
   > 한국어로 작성. 실제 성능을 테스트해보지는 않았음. 개인 프로젝트가 아닌 팀 단위 업무라고 가정할 경우, 한국어로 작성하는게 가독성을 높일 수 있다고 생각이 들어 한국어로 작성.
- [x]  `OrderTools`를 **하나의 클래스로 묶었다**. 분리해야 한다면 어떤 기준으로 나눌 것인가? (조회 vs 변경 / 주문 vs 결제) 현재 수준에서는 왜 하나로 충분한가?
   > 기본적으로는 도메인 단위로 분리한다. 단, **"항상 같이 쓰이는가"** 를 기준으로 묶는 것이 더 실용적이다.
   >
   > - **너무 크게 묶기** → 불필요한 Tool 까지 LLM 에 전달되어 토큰 낭비 발생
   > - **너무 작게 분리** → 토큰 절약이 가능하지만, 어차피 같이 쓰이는 Tool 은 항상 함께 등록하게 되어 분리 의미가 퇴색됨
   > - ex) 배달 현황 조회 시 주문 상태가 필요하므로, 분리하더라도 두 Tool 을 모두 등록해야 함
   >
   > 따라서 같이 쓰이는 Tool 끼리 묶고, 독립적으로 쓰이는 Tool 은 분리한다.  
   > 현재는 주문 관련 Tool 이 항상 함께 쓰이므로 **하나의 클래스로 충분**하다.
