# loop-play-spring-ai-agent

Spring AI 기반 배달 상담 에이전트 학습용 스타터 코드입니다.

## 개요

루퍼스 부트캠프 "Spring AI 배달 상담 에이전트" 6주 과정의 Week 1 미션 스타터 코드입니다.
`ChatClient`, System Prompt, Structured Output, Streaming, Observability 개념을 실습합니다.

## 빠른 시작

```bash
./gradlew bootRun
```

- [x]  **“이 에이전트를 그대로 프로덕션에 배포하면 어떤 사고가 날 수 있는가?”** 를 3가지 이상 작성
    1. AI 모델만 연결되어 있는 상태로 추가적인 플랫폼의 기능이 존재하지 않음.
       AI 만으로 할 수 있는 기능이 많지 않아 실제 수행할 수 있는 기능이 거의 존재하지 않음.
    2. AI 모델의 응답은 완벽히 신뢰할 수 없음. 부가적인 검증/중단 프로세스가 존재하지 않음.
       기능이 추가되면서 AI가 일을 더 많이 수행하는 경우 무한루프에 빠져 토큰 비용이 폭발적으로 증가할 수 있음.
    3. 프롬프트 기반으로만 동작하기 때문에 사용자가 프롬프트를 악의적으로 이용한다면 기존 프롬프트를 무시하고 이상동작할 가능성이 있음.

- [x]  temperature를 0.3으로 설정했다. **0.0이나 0.7이 아닌 이유는?** 데이터로 근거 제시
    - "취소 요청했는데 취소됐다고 문자는 왔는데 카드에서 돈은 아직 안 빠 졌어요. 이게 결제가 안 된 건지 환불이 된 건지 모르겠어요" 라는 분류에 모호한 문의 제시
   
      | temperature | categoryCounts | Consistency |
      |------|----------|----------|
      | 0.0 | "REFUND":5 | 1.0 |
      | 0.3 | "REFUND":5 | 1.0 |
      | 0.7 | "PAYMENT":1,"REFUND":4 | 0.8 |
      | 1.0 | "PAYMENT":4,"REFUND":1 | 0.8 |
      | 1.3 | "PAYMENT":3,"REFUND":2 | 0.6 |

    - 테스트 결과, 0.3 이상인 경우 0.7과 1.0 에서 전혀 반대의 일관성을 보임. 돈과 관련한 항목에서 응답이 일관성을 보이지 않는 건 추후 문제가 될 수 있음.
    - 적어도 0.3 이하의 속성값을 사용하는게 맞는 것으로 보이나, 이 경우 일관성이 중요한 경우라면 꼭 0.3 을 고집할 필요는 없다고 생각됩니다.
   
- [x]  구조화된 프롬프트가 단순 프롬프트보다 나은가? **어떤 상황에서는 단순 프롬프트가 더 나을 수 있는가?**
  - 간단한 역할을 주는 프롬프트("너는 배달 고객 상담 AI 야.")라도 구조화된 프롬프트로 정의.
  - 대부분의 경우, 구조화된 프롬프트가 더 나을 것으로 생각됨.
  - 다만, "이 글 번역해줘" 등 AI 가 General 하게 인지하고 있는 Domain 특화되어 있지 않은 지식을 묻는 경우, 단순 프롬프트를 사용하는게 토큰 사용량에 이점을 보일 수 있을 것으로 생각됩니다.
  - 간단한 역할만 주는 프롬프트만 사용하면 아래처럼 한국어와 다른 언어가 섞이는 경우가 발생함. 더 구조적으로 짤 수록 응답 오류를 줄일 수 있음.
    ```json
      죄송합니다. 이런 혼란스러운 상황에 직면하셨다니 불편을 드려서 미안합니다. 확인해 보겠습니다.

      1. **취소 요청 확인**: 먼저, 배달 서비스 제공업체의 취소 요청 처리 상태를 확인하겠습니다.
      2. **결제 확인**: 카드 결제 상태도 다시 한번 확인하겠습니다.
      3. **환불 절차**: 만약 환불이 완료되지 않았다면, 해당 과정을 진행하겠습니다.

      먼저, 배달 서비스 제공업체의 고객 지원 센터로 연락하시거나, 앱 내에서 취소 요청历史记录
    ```

    - a) 토큰 누적하여 표시. 응답으로 오는 Token 을 계속해서 이어붙여서 화면에 표시해줘야 함.
    - b) Markdown 렌더링 타이밍 지정.
      예를 들어 '** 응답 **' 이라는 Token 이 오면 Markdown 에서는 Bold 처리가 되어야 하지만
      마지막 * 토큰이 오기 전까지 MD 문법에 맞지 않아서 Plain Text 그대로 노출됨.
      토큰을 받을 때마다 바로 렌더링 할 것인지, Plain Text 로 표시하고 응답이 완료되면 한번에 렌더링 할 것인지 등 타이밍 정의 필요
    - c) 입력창 비활성화
      응답 스트리밍을 중에 새 메시지를 못보내게 막거나
      허용한다면 이전 응답 종료 처리 (백엔드에서도 Flux 취소 처리 해줘야 함)
    - d) 응답 완료 처리
      SSE 스펙에는 종료 이벤트가 별도로 정의되어있지 않음.
      백엔드 서버와 협의하여, 스트리밍 응답 시 어느 시점을 종료 시점으로 볼 것인지 협의 필요
      ex. complete 커스텀 이벤트를 추가하여스트리밍 응답 종료를 감지하겠다.
